    Design Thinking

    Today is #WisdomWednesday ü§©!

    Have you ever had an amazing idea, built a product around it, and found out no one wanted it? I have, plenty of times.

    Part of the issue is we started from the solution, but there is a better way.

    ‚≠ê This is a thread on Design Thinking üßµüëá

    Creating a product, whether a software, a physical thing, or a service, is all about solving a problem.

    üëâ Someone has a problem, and you offer a solution.

    But we often start with a solution, in the hope that someone wants it, and we end up solving an inexisting problem.

    This happens because, in real life, problems are not cleanly written in some kanban board.

    ‚ö†Ô∏è We often know that something is not right, and we think we know how to solve it but, actually, we don't even have a clear idea of what's wrong.

    Enter Design Thinking üëá

    Everything stems from these principles:

    • 1Ô∏è‚É£ The problem is never well defined.
    • 2Ô∏è‚É£ The problem and the solution must co-evolve together.
    • 3Ô∏è‚É£ The solution is never completed.

    And one key idea:

    ‚≠ê The human being is the most important element of the solution.

    Based on these principles, the process itself is a continuous cycle of five steps:

    • üî≠ Understand
    • üìù Define
    • üí° Ideate
    • üõ†Ô∏è Prototype
    • ‚öóÔ∏è Test

    üî≠ You start by observing people, your potential users or clients. Ask yourself:

    • ‚ùì How are they solving the problem now?
    • ‚ùì What is their critical pain in doing so?
    • ‚ùì Why is that happening?

    ü§î These questions sound familiar to you?

    Useful tools and practices:

    • üëâ Observation. Just watch people in their routine and take notes.
    • üëâ Interviews, but not with standardized questionnaires; instead, talk openly with people, and listen.
    • üëâ Ask "why" at least 5 times to get to the core of the problems.

    üìù Then you have to clearly define one problem to solve. Focus is key here.

    A good definition is actionable. It describes a situation that is easy to validate when it has been solved.

    üî¨ Go deep, you don't want to solve the superficial problem but the underlying cause.

    Here are some techniques:

    • üëâ Use a board (physical or virtual) and put a note for every potential insight or problem.
    • üëâ Make clusters, group similar problems and insights together.
    • üëâ Select (or vote on) one cluster and write a clear definition of the problem it represents.

    üí° Now it's the fun part, it's time to come up with ideas!

    This is the creative part of the process, so don't be afraid to have too many ideas. This is a rare case of quantity before quality.

    üå©Ô∏è The best tool is a brainstorming session.

    A good brainstorm has these qualities:

    • üëâ Only one person talks at a time, no interruption allowed.
    • üëâ Set a maximum time for each intervention, around 1 minute.
    • üëâ No idea is too crazy or impossible.
    • üëâ Build on top of others: instead of "no, because", just say "yes, and".

    üõ†Ô∏è Next, it's time to finally build something!

    Select one, or a few related ideas, and turn it into a prototype. You want to build a Minimum Viable Product (MVP), the smallest piece of functionality that lets you learn something new.

    üî• Don't overengineer.

    You know you are overengineering when:

    • üëâ You care about how it looks (unless that's the problem).
    • üëâ You care about how fast it is (unless that's the problem).
    • üëâ You feel sad if you have to scrap that prototype.

    ‚öóÔ∏è Finally, you have to test that prototype.

    If your problem was clearly defined, you should know what a successful evaluation looks like.

    üí¨ This is not about software testing. Test the prototype with real people.

    Things to keep in mind:

    • üëâ Shut up and watch, people should discover how the thing works by themselves.
    • üëâ Test with a variety of different profiles.
    • üëâ Write down every complaint you get, but don't mind about the proposed solutions.

    Once you reach this phase, you should have an answer to the question "is this problem solved?", which probably is:

    üî∏ Somewhat, but the problem was not so well-defined as you thought.

    That great! You just completed one cycle and learnt something new.

    üîÅ Now start over.

    ‚ö° Design Thinking applies not only to software but to any creative process.

    Whether you are planning an event, writing a book, building a physical gadget, or launching a service, you can use this strategy.

    This is not a bureaucratic methodology you have to follow, nor is a silver bullet. The key idea is this:

    üîë You're trying to solve someone's problem, so you have to understand them first.

    Everything else is just tools to help you do that.
